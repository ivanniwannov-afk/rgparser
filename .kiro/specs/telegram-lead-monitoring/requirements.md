# Документ Требований

## Введение

Система асинхронного мониторинга Telegram-чатов предназначена для автоматизации перехвата горячих лидов в Telegram-чатах. Система автоматически вступает в заданные чаты через пул юзерботов, отслеживает сообщения в реальном времени, использует LLM для квалификации потенциальных заказчиков и доставляет верифицированные лиды в личный Telegram-бот оператора.

## Глоссарий

- **Система**: Комплекс программных компонентов для мониторинга Telegram-чатов и квалификации лидов
- **Юзербот**: Telegram-аккаунт пользователя, управляемый через API (Pyrogram/Telethon) для автоматизации
- **Пул_Юзерботов**: Набор юзерботов, распределяющих нагрузку для имитации человеческого поведения
- **Модуль_Ингестии**: Компонент системы, отвечающий за автоматическое вступление в чаты
- **Очередь_Вступлений**: Асинхронная очередь задач для распределения вступлений во времени
- **Парсер_Сообщений**: Компонент для чтения и обработки сообщений из чатов в реальном времени
- **LLM_Верификатор**: Компонент для квалификации лидов через языковую модель (Claude/OpenAI)
- **Бот_Приема**: Telegram-бот для доставки квалифицированных лидов оператору
- **База_Спама**: Хранилище негативных примеров для обучения LLM фильтрации
- **Лид**: Сообщение в чате, содержащее потенциальный запрос на услуги
- **Квалифицированный_Лид**: Лид, верифицированный LLM как реальный запрос заказчика
- **Триггерное_Слово**: Ключевое слово, указывающее на потенциальный лид

## Требования

### Требование 1: Прием Списка Чатов

**User Story:** Как оператор, я хочу отправить боту список ссылок на чаты, чтобы система автоматически вступила в них для мониторинга.

#### Критерии Приемки

1. КОГДА оператор отправляет список ссылок формата t.me/chatname или @username, Модуль_Ингестии ДОЛЖЕН принять и сохранить список
2. Модуль_Ингестии ДОЛЖЕН валидировать формат каждой ссылки перед сохранением
3. КОГДА ссылка имеет невалидный формат, Модуль_Ингестии ДОЛЖЕН вернуть оператору сообщение об ошибке с указанием невалидной ссылки
4. Модуль_Ингестии ДОЛЖЕН сохранять список чатов в базе данных со статусом "ожидает вступления"

### Требование 2: Распределение Чатов Между Юзерботами

**User Story:** Как система, я хочу равномерно распределить чаты между юзерботами, чтобы избежать перегрузки одного аккаунта.

#### Критерии Приемки

1. КОГДА список чатов принят, Модуль_Ингестии ДОЛЖЕН распределить чаты между доступными юзерботами из Пул_Юзерботов
2. Модуль_Ингестии ДОЛЖЕН учитывать текущую нагрузку каждого юзербота при распределении
3. Модуль_Ингестии ДОЛЖЕН ограничивать количество вступлений для одного юзербота до 10 чатов в сутки
4. КОГДА юзербот достиг дневного лимита вступлений, Модуль_Ингестии ДОЛЖЕН исключить его из распределения до следующих суток

### Требование 3: Асинхронная Очередь Вступлений

**User Story:** Как система, я хочу вступать в чаты с рандомизированными задержками, чтобы имитировать человеческое поведение и избежать блокировки.

#### Критерии Приемки

1. Очередь_Вступлений ДОЛЖНА добавлять задачи вступления с рандомизированной задержкой от 300 до 1800 секунд между вступлениями одного юзербота
2. КОГДА задача вступления создана, Очередь_Вступлений ДОЛЖНА назначить время выполнения с учетом задержки
3. Очередь_Вступлений ДОЛЖНА обрабатывать задачи асинхронно без блокировки других компонентов системы
4. КОГДА наступает время выполнения задачи, Очередь_Вступлений ДОЛЖНА передать задачу соответствующему юзерботу

### Требование 4: Выполнение Вступления в Чат

**User Story:** Как юзербот, я хочу вступить в назначенный чат, чтобы начать мониторинг сообщений.

#### Критерии Приемки

1. КОГДА юзербот получает задачу вступления, Модуль_Ингестии ДОЛЖЕН выполнить вступление через Telegram API
2. КОГДА вступление успешно, Модуль_Ингестии ДОЛЖЕН обновить статус чата на "активный мониторинг"
3. ЕСЛИ вступление не удалось, Модуль_Ингестии ДОЛЖЕН записать причину ошибки и обновить статус чата на "ошибка вступления"
4. ЕСЛИ чат требует подтверждения администратора, Модуль_Ингестии ДОЛЖЕН обновить статус чата на "ожидает подтверждения"

### Требование 5: Обработка Антибот-Защиты

**User Story:** Как система, я хочу обрабатывать простые антибот-механизмы при вступлении, чтобы успешно присоединяться к защищенным чатам.

#### Критерии Приемки

1. КОГДА при вступлении появляется inline-кнопка приветственного бота, Модуль_Ингестии ДОЛЖЕН нажать на кнопку
2. Модуль_Ингестии ДОЛЖЕН логировать все случаи обнаружения антибот-защиты с типом защиты
3. ЕСЛИ антибот-защита не может быть обработана автоматически, Модуль_Ингестии ДОЛЖЕН уведомить оператора и обновить статус чата на "требует ручной обработки"
4. Модуль_Ингестии ДОЛЖЕН ожидать до 60 секунд после нажатия кнопки перед подтверждением успешного вступления

### Требование 6: Парсинг Сообщений в Реальном Времени

**User Story:** Как система, я хочу читать все сообщения из активных чатов в реальном времени, чтобы не пропустить потенциальные лиды.

#### Критерии Приемки

1. КОГДА чат имеет статус "активный мониторинг", Парсер_Сообщений ДОЛЖЕН подписаться на получение новых сообщений через Telegram API
2. Парсер_Сообщений ДОЛЖЕН обрабатывать входящие сообщения асинхронно без блокировки других чатов
3. Парсер_Сообщений ДОЛЖЕН извлекать из сообщения текст, имя отправителя, ссылку на профиль отправителя и название чата
4. КОГДА сообщение получено, Парсер_Сообщений ДОЛЖЕН передать его на проверку триггерных слов в течение 1 секунды

### Требование 7: Фильтрация по Триггерным Словам

**User Story:** Как система, я хочу предварительно фильтровать сообщения по триггерным словам, чтобы отправлять на LLM-верификацию только релевантные сообщения.

#### Критерии Приемки

1. Парсер_Сообщений ДОЛЖЕН хранить список триггерных слов в конфигурации системы
2. КОГДА сообщение содержит хотя бы одно триггерное слово, Парсер_Сообщений ДОЛЖЕН передать его в LLM_Верификатор
3. КОГДА сообщение не содержит триггерных слов, Парсер_Сообщений ДОЛЖЕН отбросить сообщение без дальнейшей обработки
4. Парсер_Сообщений ДОЛЖЕН выполнять проверку триггерных слов без учета регистра

### Требование 8: LLM-Верификация Лидов

**User Story:** Как система, я хочу использовать LLM для квалификации потенциальных лидов, чтобы отделить реальные запросы от спама и обсуждений.

#### Критерии Приемки

1. КОГДА сообщение передано на верификацию, LLM_Верификатор ДОЛЖЕН отправить запрос к LLM API (Claude или OpenAI) с текстом сообщения
2. LLM_Верификатор ДОЛЖЕН включать в системный промпт инструкции по определению реальных запросов на услуги
3. LLM_Верификатор ДОЛЖЕН получить от LLM бинарный ответ: "квалифицированный лид" или "не лид"
4. КОГДА LLM классифицирует сообщение как квалифицированный лид, LLM_Верификатор ДОЛЖЕН передать его в Бот_Приема
5. ЕСЛИ запрос к LLM API не удался, LLM_Верификатор ДОЛЖЕН повторить попытку до 3 раз с экспоненциальной задержкой
6. LLM_Верификатор ДОЛЖЕН обрабатывать запросы через внутреннюю очередь с ограничением concurrency не более 10 одновременных запросов к API для защиты от ошибки 429 Rate Limit

### Требование 9: Использование Негативных Примеров в LLM-Промпте

**User Story:** Как система, я хочу использовать примеры спама из базы данных в промпте LLM, чтобы улучшить точность фильтрации.

#### Критерии Приемки

1. КОГДА LLM_Верификатор формирует запрос к LLM, он ДОЛЖЕН извлечь до 20 последних примеров из База_Спама
2. LLM_Верификатор ДОЛЖЕН включить негативные примеры в системный промпт в формате few-shot learning
3. ГДЕ База_Спама пуста, LLM_Верификатор ДОЛЖЕН использовать промпт без негативных примеров
4. LLM_Верификатор ДОЛЖЕН обновлять кэш негативных примеров каждые 60 секунд

### Требование 10: Доставка Квалифицированных Лидов

**User Story:** Как оператор, я хочу получать квалифицированные лиды в Telegram-бот, чтобы оперативно связаться с потенциальными заказчиками.

#### Критерии Приемки

1. КОГДА лид квалифицирован, Бот_Приема ДОЛЖЕН отправить оператору сообщение с текстом лида
2. Бот_Приема ДОЛЖЕН включить в сообщение ссылку на профиль отправителя в формате @username или t.me/userid
3. Бот_Приема ДОЛЖЕН включить в сообщение название чата, из которого получен лид
4. Бот_Приема ДОЛЖЕН добавить inline-кнопки "В блок" и "Спам" под каждым сообщением с лидом

### Требование 11: Динамическая Фильтрация Спама

**User Story:** Как оператор, я хочу помечать ложные срабатывания как спам, чтобы система обучалась и не показывала похожие сообщения в будущем.

#### Критерии Приемки

1. КОГДА оператор нажимает кнопку "Спам", Бот_Приема ДОЛЖЕН сохранить текст помеченного сообщения в База_Спама
2. Бот_Приема ДОЛЖЕН добавить метку времени к записи в База_Спама
3. Бот_Приема ДОЛЖЕН подтвердить оператору успешное добавление в базу спама
4. КОГДА оператор нажимает кнопку "В блок", Бот_Приема ДОЛЖЕН сохранить отправителя в список заблокированных и добавить сообщение в База_Спама

### Требование 12: Управление Пулом Юзерботов

**User Story:** Как администратор, я хочу добавлять и удалять юзерботов из пула, чтобы масштабировать систему и заменять заблокированные аккаунты.

#### Критерии Приемки

1. Система ДОЛЖНА предоставлять интерфейс для добавления нового юзербота с указанием session-файла
2. КОГДА юзербот добавлен, Система ДОЛЖНА проверить валидность сессии через Telegram API
3. ЕСЛИ сессия валидна, Система ДОЛЖНА добавить юзербота в Пул_Юзерботов со статусом "активен"
4. Система ДОЛЖНА предоставлять интерфейс для деактивации юзербота с сохранением его данных

### Требование 13: Мониторинг Состояния Юзерботов

**User Story:** Как администратор, я хочу отслеживать состояние юзерботов, чтобы своевременно обнаруживать блокировки и проблемы.

#### Критерии Приемки

1. Система ДОЛЖНА проверять доступность каждого юзербота каждые 300 секунд
2. ЕСЛИ юзербот не отвечает на проверку, Система ДОЛЖНА обновить его статус на "недоступен"
3. КОГДА юзербот получает ошибку "USER_DEACTIVATED_BAN" от Telegram API, Система ДОЛЖНА обновить его статус на "заблокирован"
4. Система ДОЛЖНА уведомлять администратора при обнаружении заблокированного или недоступного юзербота

### Требование 14: Логирование Активности Системы

**User Story:** Как администратор, я хочу видеть логи всех операций системы, чтобы диагностировать проблемы и анализировать эффективность.

#### Критерии Приемки

1. Система ДОЛЖНА логировать все вступления в чаты с указанием юзербота, чата и результата
2. Система ДОЛЖНА логировать все обращения к LLM API с указанием текста запроса и ответа
3. Система ДОЛЖНА логировать все доставленные лиды с указанием чата и времени
4. Система ДОЛЖНА логировать все ошибки с указанием компонента, типа ошибки и stack trace

### Требование 15: Ограничение Режима Работы Юзерботов

**User Story:** Как система, я хочу использовать юзерботов только в режиме чтения, чтобы минимизировать риск блокировки.

#### Критерии Приемки

1. Парсер_Сообщений ДОЛЖЕН использовать юзерботов только для чтения сообщений без отправки
2. ГДЕ требуется вступление в чат, Модуль_Ингестии ДОЛЖЕН выполнять только операцию join без дополнительных действий
3. Система ДОЛЖНА запрещать отправку сообщений через юзерботов после успешного вступления
4. ЕСЛИ компонент системы пытается отправить сообщение через юзербота, Система ДОЛЖНА заблокировать операцию и записать предупреждение в лог
5. Парсер_Сообщений ДОЛЖЕН работать в режиме абсолютной невидимости без отправки read receipts (отметок о прочтении) и без трансляции статуса "Онлайн"

### Требование 16: Обработка Ограничений Telegram API

**User Story:** Как система, я хочу соблюдать лимиты Telegram API, чтобы избежать временных блокировок и ошибок FloodWait.

#### Критерии Приемки

1. КОГДА Система получает ошибку FloodWait от Telegram API, она ДОЛЖНА приостановить операции затронутого юзербота на указанное в ошибке время
2. Система ДОЛЖНА ограничивать количество запросов к Telegram API до 20 запросов в секунду на один юзербот
3. КОГДА юзербот приостановлен из-за FloodWait, Система ДОЛЖНА перераспределить его задачи на другие доступные юзерботы
4. Система ДОЛЖНА возобновить работу юзербота автоматически после истечения времени FloodWait

### Требование 17: Хранение Конфигурации Системы

**User Story:** Как администратор, я хочу хранить конфигурацию системы в файле, чтобы легко изменять параметры без перезапуска.

#### Критерии Приемки

1. Система ДОЛЖНА загружать конфигурацию из файла формата JSON или YAML при запуске
2. Конфигурация ДОЛЖНА включать список триггерных слов, лимиты вступлений, диапазоны задержек и параметры LLM API
3. Система ДОЛЖНА валидировать конфигурацию при загрузке и сообщать об ошибках
4. ГДЕ конфигурационный файл отсутствует, Система ДОЛЖНА создать файл с параметрами по умолчанию

### Требование 18: Персистентность Состояния Системы

**User Story:** Как система, я хочу сохранять свое состояние в базе данных, чтобы продолжить работу после перезапуска без потери данных.

#### Критерии Приемки

1. Система ДОЛЖНА сохранять состояние всех чатов (статус, назначенный юзербот, время последнего сообщения) в базе данных
2. Система ДОЛЖНА сохранять состояние всех задач в Очередь_Вступлений в базе данных
3. КОГДА Система запускается, она ДОЛЖНА восстановить состояние из базы данных
4. Система ДОЛЖНА использовать транзакции для обеспечения консистентности данных при сохранении состояния

### Требование 19: Обработка Дубликатов Лидов

**User Story:** Как оператор, я хочу получать каждый уникальный лид только один раз в сутки, чтобы избежать дублирования информации от кросс-чатового спама.

#### Критерии Приемки

1. Система ДОЛЖНА вычислять хэш каждого сообщения ТОЛЬКО на основе очищенного текста (без спецсимволов, эмодзи, ссылок и пробелов), игнорируя отправителя
2. КОГДА сообщение обработано, Система ДОЛЖНА сохранить его хэш в базе данных с меткой времени
3. КОГДА новое сообщение имеет хэш, уже существующий в базе данных и созданный менее 24 часов назад, Система ДОЛЖНА отбросить сообщение без обработки
4. Система ДОЛЖНА удалять хэши сообщений старше 24 часов для обеспечения доставки одного одинакового сообщения максимум раз в сутки

### Требование 20: Graceful Shutdown

**User Story:** Как администратор, я хочу корректно останавливать систему, чтобы не потерять данные и не оставить задачи в неопределенном состоянии.

#### Критерии Приемки

1. КОГДА Система получает сигнал остановки (SIGTERM или SIGINT), она ДОЛЖНА прекратить принимать новые задачи
2. Система ДОЛЖНА завершить все активные задачи в течение 30 секунд после получения сигнала остановки
3. Система ДОЛЖНА сохранить состояние всех незавершенных задач в базе данных перед остановкой
4. ЕСЛИ задачи не завершились за 30 секунд, Система ДОЛЖНА принудительно остановиться с сохранением текущего состояния
