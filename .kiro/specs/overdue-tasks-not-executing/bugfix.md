# Документ требований к исправлению бага

## Введение

Этот документ описывает требования к исправлению критического бага, при котором просроченные задачи не выполняются при перезапуске системы. В настоящее время, если задача запланирована с задержкой и система останавливается до выполнения задачи, задача остается в статусе 'pending' навсегда после перезапуска, даже если её запланированное время уже прошло. Это предотвращает выполнение задач при любом простое системы, что является критической проблемой надежности.

## Анализ бага

### Текущее поведение (Дефект)

1.1 КОГДА задача создается с задержкой 120 секунд и добавляется в очередь ТО система корректно планирует задачу

1.2 КОГДА система останавливается (Ctrl+C) до запланированного времени ТО задача остается в статусе 'pending' в базе данных

1.3 КОГДА запланированное время проходит пока система остановлена (задача становится просроченной) ТО временная метка scheduled_at задачи становится меньше текущего времени

1.4 КОГДА система перезапускается и загружает pending задачи через load_pending_tasks() ТО просроченные задачи загружаются но остаются в статусе 'pending'

1.5 КОГДА вызывается get_next_task() после перезапуска с просроченными задачами ТО просроченные задачи НЕ возвращаются для выполнения

1.6 КОГДА запускается check_status.py ТО он показывает "ПРОСРОЧЕНО на X минут" но задача никогда не выполняется

1.7 КОГДА просроченная задача остается необработанной ТО она остается в статусе 'pending' бесконечно

1.8 КОГДА система выполняет операции (присоединение к чатам, обработка сообщений) ТО логи активности НЕ записываются в таблицу activity_logs

1.9 КОГДА пользователь проверяет логи через дашборд или check_status.py ТО отображается пустой список или старые логи

1.10 КОГДА происходят ошибки или успешные операции ТО нет записей в базе данных для отладки и мониторинга

### Ожидаемое поведение (Правильное)

2.1 КОГДА система перезапускается и загружает pending задачи через load_pending_tasks() ТО все pending задачи включая просроченные ДОЛЖНЫ быть загружены в память

2.2 КОГДА вызывается get_next_task() и есть задачи с scheduled_at <= текущее время ТО система ДОЛЖНА вернуть просроченную задачу немедленно

2.3 КОГДА просроченная задача возвращается get_next_task() ТО система ДОЛЖНА выполнить её немедленно без дополнительной задержки

2.4 КОГДА просроченная задача завершает выполнение ТО система ДОЛЖНА обновить её статус на 'completed' или 'failed' в зависимости от результата выполнения

2.5 КОГДА запускается check_status.py после перезапуска системы ТО ранее просроченные задачи ДОЛЖНЫ показываться как 'completed' или 'failed', а не 'pending'

2.6 КОГДА система выполняет любую операцию (присоединение, обработка сообщений, ошибки) ТО запись ДОЛЖНА добавляться в таблицу activity_logs с правильными полями (timestamp, level, component, message)

2.7 КОГДА пользователь проверяет логи ТО система ДОЛЖНА показывать актуальные записи всех операций

2.8 КОГДА происходит критическая операция или ошибка ТО лог ДОЛЖЕН содержать достаточно информации для отладки

2.9 КОГДА проводится полный аудит логики приложения ТО ВСЕ компоненты (очередь задач, загрузка задач, выполнение задач, логирование) ДОЛЖНЫ быть проверены на корректность работы

2.10 КОГДА обнаруживаются проблемы в логике ТО они ДОЛЖНЫ быть задокументированы и исправлены

### Неизменное поведение (Предотвращение регрессии)

3.1 КОГДА задача запланирована на будущее время и система НЕ перезапускается ТО система ДОЛЖНА ПРОДОЛЖАТЬ выполнять задачу в запланированное время

3.2 КОГДА наступает время scheduled_at задачи пока система работает ТО система ДОЛЖНА ПРОДОЛЖАТЬ выполнять задачу немедленно

3.3 КОГДА задача успешно завершается во время нормальной работы ТО система ДОЛЖНА ПРОДОЛЖАТЬ обновлять её статус на 'completed'

3.4 КОГДА задача проваливается во время нормальной работы ТО система ДОЛЖНА ПРОДОЛЖАТЬ обновлять её статус на 'failed'

3.5 КОГДА в очереди несколько задач с разными запланированными временами ТО система ДОЛЖНА ПРОДОЛЖАТЬ обрабатывать их в правильном порядке на основе scheduled_at

3.6 КОГДА новые задачи добавляются в очередь пока система работает ТО система ДОЛЖНА ПРОДОЛЖАТЬ обрабатывать их корректно
