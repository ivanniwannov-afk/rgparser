# Технический дизайн исправления бага с просроченными задачами

## Обзор

Этот дизайн решает критический баг, при котором просроченные задачи присоединения (задачи с `scheduled_at <= текущее_время`) не выполняются после перезапуска системы. Баг возникает из-за того, что `load_pending_tasks()` в `JoinQueue` фильтрует задачи по `created_at > cutoff_time` (1 час назад), что означает что задачи созданные более часа назад никогда не загружаются в очередь, даже если они просрочены всего на несколько минут.

Исправление минимальное и хирургическое: изменить `load_pending_tasks()` чтобы загружать ВСЕ pending задачи или фильтровать по `scheduled_at` вместо `created_at`. Это гарантирует что просроченные задачи выполняются немедленно, сохраняя все существующее поведение для будущих задач.

**Дополнительно**: Необходимо провести полный аудит логики приложения и исправить проблему с записью логов в activity_logs.

## Глоссарий

- **Bug_Condition (C)**: Условие которое вызывает баг - когда время `scheduled_at` задачи в прошлом (просрочена) и вызывается `load_pending_tasks()` или `get_next_task()`
- **Property (P)**: Желаемое поведение - просроченные задачи должны загружаться и возвращаться немедленно для выполнения без задержки
- **Preservation**: Существующее поведение для будущих задач и нормальной обработки задач которое должно остаться неизменным
- **get_next_task()**: Метод в `src/ingestion/join_queue.py` (строка 145) который извлекает следующую задачу готовую к выполнению
- **load_pending_tasks()**: Метод в `src/ingestion/join_queue.py` (строка 113) который загружает pending задачи из базы данных при старте
- **Просроченная задача**: Задача где `scheduled_at <= текущее_время`, то есть её запланированное время выполнения уже прошло
- **activity_logs**: Таблица в базе данных для хранения логов активности системы

## Детали бага

### Условие бага

Баг проявляется когда система перезапускается и в базе данных есть pending задачи чье время `scheduled_at` уже прошло (просроченные задачи). Метод `load_pending_tasks()` НЕ загружает эти задачи если они были созданы более часа назад, даже если они просрочены всего на несколько минут.

**Формальная спецификация:**
```
FUNCTION isBugCondition(task)
  INPUT: task типа JoinTask
  OUTPUT: boolean
  
  RETURN task.scheduled_at <= текущее_время
         AND task.status == 'pending'
         AND task.created_at < (текущее_время - 1 час)
         AND load_pending_tasks() вызывается
END FUNCTION
```

### Анализ корневой причины

**КОРНЕВАЯ ПРИЧИНА:**

В методе `load_pending_tasks()` (строки 113-143 файла `join_queue.py`), фильтр загрузки использует `created_at`:

```python
# Вычисляем время отсечки (1 час назад)
cutoff_time = datetime.now(timezone.utc) - timedelta(hours=1)
cutoff_str = cutoff_time.isoformat()

cursor = await db.execute(
    """SELECT id, userbot_id, chat_id, scheduled_at
       FROM join_tasks
       WHERE status = 'pending' AND created_at > ?
       ORDER BY scheduled_at ASC""",
    (cutoff_str,)
)
```

Это означает что если задача была создана более 1 часа назад но просрочена всего на несколько минут (например, создана 65 минут назад, запланирована на 5 минут назад), она НЕ будет загружена в очередь вообще!

**Пример проблемы:**
- Задача создана в 09:00
- Задача запланирована на 10:00 (задержка 1 час)
- Система остановлена в 09:30
- Система перезапущена в 10:05
- Задача просрочена на 5 минут
- НО задача создана 65 минут назад (> 1 час)
- `cleanup_old_tasks()` помечает её как 'failed'
- Задача НИКОГДА не выполняется

### Примеры

- **Пример 1**: Задача создана в 10:00, запланирована на 10:02 (задержка 2 минуты). Система останавливается в 10:01. Система перезапускается в 10:05. Задача просрочена на 3 минуты. Если задача создана < 1 часа назад, она загружается и `get_next_task()` должен вернуть её немедленно. Если создана > 1 часа назад, она помечается как failed через `cleanup_old_tasks()`.

- **Пример 2**: Задача создана в 09:00, запланирована на 10:00 (задержка 1 час). Система останавливается в 09:30. Система перезапускается в 10:05. Задача просрочена на 5 минут но создана 65 минут назад. `cleanup_old_tasks()` помечает её как failed потому что `created_at < cutoff_time`.

- **Пример 3**: Задача создана в 10:00, запланирована на 10:02. Система работает нормально. В 10:02 `get_next_task()` возвращает задачу немедленно (РАБОТАЕТ ПРАВИЛЬНО).

- **Граничный случай**: Несколько просроченных задач с разными `scheduled_at`. Они должны обрабатываться в порядке `scheduled_at` (самая ранняя первой).

### Проблема с логами

**ВТОРАЯ ПРОБЛЕМА**: Логи активности не записываются в таблицу `activity_logs`.

Необходимо проверить:

## Ожидаемое поведение

### Требования к сохранению

**Неизменное поведение:**
- Задачи запланированные на будущее время должны продолжать ждать до запланированного времени перед выполнением
- Задачи которые становятся готовыми пока система работает должны продолжать выполняться в запланированное время
- Обновления статуса задач (pending → processing → completed/failed) должны остаться неизменными
- Упорядочивание приоритетной очереди по `scheduled_at` должно остаться неизменным
- Логика очистки задач старше 1 часа должна остаться неизменной

**Область действия:**
Все входные данные которые НЕ связаны с просроченными задачами (задачи где `scheduled_at > текущее_время`) должны быть полностью не затронуты этим исправлением. Это включает:
- Будущие задачи которые загружаются при старте
- Новые задачи добавленные пока система работает
- Задачи которые становятся готовыми пока ждут в очереди
- Нормальный поток выполнения задач во время непрерывной работы

## Гипотеза корневой причины

На основе описания бага и анализа кода, есть несколько потенциальных корневых причин:

1. **Слишком строгий фильтр загрузки** (ОСНОВНАЯ ПРИЧИНА): Метод `load_pending_tasks()` фильтрует по `created_at > cutoff_time` (1 час назад), что означает что задачи созданные более 1 часа назад никогда не загружаются в очередь, даже если они просрочены всего немного.

2. **Проблема с таймингом в get_next_task()**: Может быть состояние гонки или проблема с таймингом где просроченные задачи не сразу распознаются как готовые к выполнению, хотя логика кода выглядит правильной.

3. **Таймаут пустой очереди**: Таймаут в 1 секунду в `get_next_task()` может привести к пропуску просроченных задач если очередь кажется пустой на мгновение.

4. **Конкурентное добавление задач**: Если задачи добавляются в очередь пока `get_next_task()` обрабатывает, могут быть проблемы с упорядочиванием приоритетной очереди.

5. **Логирование не работает**: `ActivityLogger` может не записывать в базу данных из-за ошибок или неправильной конфигурации.

Наиболее вероятная проблема - #1: фильтр загрузки слишком строгий и предотвращает загрузку просроченных задач вообще.

## Свойства корректности

Property 1: Bug Condition - Просроченные задачи выполняются немедленно

_Для любой_ задачи присоединения где `scheduled_at <= текущее_время` и `status == 'pending'`, исправленная функция `load_pending_tasks()` ДОЛЖНА загрузить эту задачу, и `get_next_task()` ДОЛЖЕН вернуть эту задачу немедленно без задержки, и задача ДОЛЖНА быть выполнена через `_process_join_queue()`.

**Валидирует: Требования 2.1, 2.2, 2.3, 2.4, 2.5**

Property 2: Preservation - Планирование будущих задач

_Для любой_ задачи присоединения где `scheduled_at > текущее_время`, исправленный код ДОЛЖЕН производить точно такое же поведение как оригинальный код, ожидая до запланированного времени перед возвратом задачи для выполнения.

**Валидирует: Требования 3.1, 3.2, 3.3, 3.4, 3.5, 3.6**

## Реализация исправления

### Требуемые изменения

На основе анализа корневой причины, исправление должно решить проблему фильтра загрузки:

**Файл**: `src/ingestion/join_queue.py`

**Метод**: `load_pending_tasks()` (строки 113-143)

**Конкретные изменения**:

1. **Изменить логику фильтра загрузки**: Изменить запрос чтобы загружать ВСЕ pending задачи независимо от `created_at`, ИЛИ изменить фильтр чтобы проверять `scheduled_at` вместо `created_at`:

   **Вариант A (Рекомендуется)**: Загружать все pending задачи
   ```python
   cursor = await db.execute(
       """SELECT id, userbot_id, chat_id, scheduled_at
          FROM join_tasks
          WHERE status = 'pending'
          ORDER BY scheduled_at ASC"""
   )
   ```

   **Вариант B**: Фильтровать по `scheduled_at` вместо `created_at`
   ```python
   # Загружать задачи запланированные в разумном окне (например, последние 24 часа)
   cutoff_time = datetime.now(timezone.utc) - timedelta(hours=24)
   cutoff_str = cutoff_time.isoformat()
   
   cursor = await db.execute(
       """SELECT id, userbot_id, chat_id, scheduled_at
          FROM join_tasks
          WHERE status = 'pending' AND scheduled_at > ?
          ORDER BY scheduled_at ASC""",
       (cutoff_str,)
   )
   ```

2. **Сохранить cleanup_old_tasks() отдельно**: Метод `cleanup_old_tasks()` должен продолжать помечать очень старые задачи как failed (created > 1 час назад), но это должно быть основано на `created_at`, а не `scheduled_at`. Это предотвращает накопление действительно заброшенных задач.

3. **Проверить логику get_next_task()**: Убедиться что логика обнаружения просроченных задач правильная (она выглядит правильной, но нужно проверить тестами).

4. **Исправить логирование**: Проверить все вызовы `ActivityLogger` и убедиться что логи записываются в базу данных.

**Обоснование**: Проблема в том что `load_pending_tasks()` использует `created_at` для фильтрации, но задача может быть создана давно и запланирована на недавнее выполнение. Мы должны либо загружать все pending задачи (позволяя `cleanup_old_tasks()` обрабатывать действительно старые), либо фильтровать по `scheduled_at` чтобы ловить недавно запланированные задачи независимо от времени создания.

### Рекомендуемый подход

Использовать Вариант A (загружать все pending задачи) потому что:
- Более простая логика с меньшим количеством граничных случаев
- `cleanup_old_tasks()` уже обрабатывает действительно старые задачи (created > 1 час назад)
- Просроченные задачи которые не "старые" будут загружены и выполнены немедленно
- Нет риска пропустить задачи из-за временных окон

## Стратегия тестирования

### Подход к валидации

Стратегия тестирования следует двухфазному подходу: сначала выявить контрпримеры демонстрирующие баг на неисправленном коде, затем проверить что исправление работает корректно и сохраняет существующее поведение.

### Исследовательская проверка условия бага

**Цель**: Выявить контрпримеры демонстрирующие баг ДО реализации исправления. Подтвердить или опровергнуть анализ корневой причины.

**План тестирования**: Создать тест который симулирует точный сценарий бага:
1. Создать задачу с короткой задержкой (например, 5 секунд)
2. Подождать пока задача станет просроченной (подождать 10 секунд)
3. Симулировать перезапуск системы создав новый экземпляр JoinQueue
4. Вызвать `load_pending_tasks()` чтобы загрузить просроченную задачу
5. Вызвать `get_next_task()` и проверить что она возвращает задачу немедленно (не None)
6. Измерить затраченное время - должно быть < 1 секунды

Запустить этот тест на НЕИСПРАВЛЕННОМ коде чтобы наблюдать сбой.

**Тестовые случаи**:
1. **Просроченная задача не загружена**: Создать задачу, подождать пока она станет просроченной, перезапустить очередь, проверить что задача НЕ загружена (провалится на неисправленном коде если created > 1 час назад)
2. **Просроченная задача не возвращена**: Создать задачу, подождать пока она станет просроченной, перезапустить очередь, вызвать `get_next_task()`, проверить что возвращает None или зависает (провалится на неисправленном коде)
3. **Несколько просроченных задач**: Создать 3 задачи с разными scheduled times, все просрочены, проверить что возвращаются в правильном порядке (провалится на неисправленном коде)
4. **Граничный случай - только что просрочена**: Создать задачу запланированную 1 секунду назад, проверить что возвращается немедленно (может провалиться на неисправленном коде из-за тайминга)

**Ожидаемые контрпримеры**:
- `load_pending_tasks()` возвращает 0 для задач created > 1 час назад, даже если просрочены всего немного
- `get_next_task()` возвращает None или зависает бесконечно для просроченных задач
- Задачи остаются в статусе 'pending' навсегда после перезапуска

### Проверка исправления

**Цель**: Проверить что для всех входных данных где выполняется условие бага, исправленная функция производит ожидаемое поведение.

**Псевдокод:**
```
FOR ALL task WHERE isBugCondition(task) DO
  queue := JoinQueue()
  await queue.load_pending_tasks()
  result := await queue.get_next_task()
  ASSERT result IS NOT None
  ASSERT result.task_id == task.task_id
  ASSERT result.scheduled_at <= текущее_время
END FOR
```

### Проверка сохранения

**Цель**: Проверить что для всех входных данных где условие бага НЕ выполняется, исправленная функция производит тот же результат что и оригинальная функция.

**Псевдокод:**
```
FOR ALL task WHERE NOT isBugCondition(task) DO
  # task.scheduled_at > текущее_время (будущая задача)
  queue := JoinQueue()
  await queue.add_task(task)
  
  start_time := текущее_время
  result := await queue.get_next_task()
  end_time := текущее_время
  
  ASSERT result IS NOT None
  ASSERT result.task_id == task.task_id
  ASSERT (end_time - start_time) >= (task.scheduled_at - start_time)
END FOR
```
