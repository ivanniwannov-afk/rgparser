# Технический дизайн исправления бага с задержками задач

## Обзор

Система создает задачи присоединения к чатам с задержками 420+ минут вместо настроенных 60-104 секунд. Анализ показал, что проблема не в самом создании задач (они создаются с правильными задержками), а в том, что задачи не добавляются в очередь выполнения после создания. Это приводит к накоплению задач в базе данных, которые при перезапуске системы загружаются как старые задачи с большими временными смещениями.

Стратегия исправления: обеспечить, чтобы каждая созданная задача немедленно добавлялась в очередь выполнения (`join_queue`), и исправить отображение статистики задержек в утилите проверки статуса.

## Глоссарь

- **Bug_Condition (C)**: Условие, при котором задачи создаются в базе данных, но не добавляются в очередь выполнения
- **Property (P)**: Желаемое поведение - каждая созданная задача должна быть добавлена в очередь и выполнена с задержкой из диапазона join_delay_min..join_delay_max
- **Preservation**: Существующее поведение создания задач с правильными задержками, которое должно остаться неизменным
- **enqueue_join_tasks()**: Метод в `src/ingestion/ingestion_module.py`, который создает записи задач в таблице `join_tasks`
- **join_queue**: Очередь выполнения задач в памяти, управляемая классом `JoinQueue`
- **_process_pending_chats()**: Метод в `main.py`, который обрабатывает ожидающие чаты и создает задачи присоединения
- **scheduled_at**: Временная метка, когда задача должна быть выполнена (created_at + delay)
- **delay**: Случайная задержка в секундах между join_delay_min и join_delay_max

## Детали бага

### Условие бага

Баг проявляется, когда система создает задачи присоединения через метод `_process_pending_chats()` в `main.py`. Метод `enqueue_join_tasks()` корректно создает записи в базе данных с правильными задержками (60-104 секунды), но эти задачи не добавляются в очередь выполнения `join_queue`. В результате задачи остаются в базе со статусом 'pending' и никогда не выполняются. При перезапуске системы метод `load_pending_tasks()` загружает все накопившиеся задачи, включая старые, что создает видимость огромных задержек.

**Формальная спецификация:**
```
FUNCTION isBugCondition(task_creation_event)
  INPUT: task_creation_event содержит {distribution, ingestion_module, join_queue}
  OUTPUT: boolean
  
  RETURN task_created_in_database(task_creation_event)
         AND NOT task_added_to_queue(task_creation_event.join_queue)
         AND task_creation_path == "_process_pending_chats"
END FUNCTION
```

### Примеры

- **Пример 1**: Пользователь добавляет канал через дашборд → `_process_pending_chats()` создает задачу в базе с delay=75 секунд → задача НЕ добавляется в `join_queue` → задача остается в базе со статусом 'pending' → при перезапуске через 7 часов задача загружается и выглядит как задержка 420+ минут

- **Пример 2**: Система запускается с 5 ожидающими чатами → `_process_pending_chats()` создает 5 задач с delays 60-104 секунды → задачи НЕ добавляются в очередь → через день пользователь запускает `check_status.bat` → видит задачи с "задержками" 1440+ минут (24 часа)

- **Пример 3**: `debug_task_creation.bat` создает задачи напрямую через `enqueue_join_tasks()` с delays 9, 19, 34 секунды → эти задачи корректны → но `check_status.bat` также показывает Task #9 с delay 489 минут → это старая задача из предыдущего запуска, которая не была добавлена в очередь

- **Край случай**: Пользователь запускает `clear_pending_tasks.bat` → все задачи удаляются → добавляет новый канал → задача создается правильно, но не добавляется в очередь → система перезапускается через 1 минуту → задача загружается и выглядит как delay 60-104 секунды (правильно), но если перезапуск через час → delay выглядит как 3600+ секунд

## Ожидаемое поведение

### Требования к сохранению

**Неизменное поведение:**
- Метод `enqueue_join_tasks()` должен продолжать создавать задачи в базе данных с правильными задержками (join_delay_min..join_delay_max секунд)
- Распределение чатов между юзерботами через `distribute_chats()` должно продолжать использовать round-robin балансировку
- Обработка задач очередью присоединения должна продолжать выполняться в запланированное время
- Загрузка конфигурации из config.json должна продолжать работать корректно
- Равномерное распределение задач между доступными юзерботами должно сохраниться
- Исключение юзерботов, достигших дневного лимита, из распределения должно продолжать работать
- Обновление статуса задач (completed/failed) в базе данных должно продолжать работать правильно

**Область действия:**
Все входные данные, которые НЕ связаны с путем создания задач через `_process_pending_chats()`, должны быть полностью не затронуты исправлением. Это включает:
- Создание задач через другие пути (если они существуют)
- Выполнение задач процессором очереди
- Обновление статусов юзерботов и чатов
- Логирование активности

## Гипотеза корневой причины

На основе анализа кода и описания бага, наиболее вероятные причины:

1. **Отсутствие добавления в очередь после создания**: Метод `_process_pending_chats()` в `main.py` вызывает `enqueue_join_tasks()` для создания записей в базе, но не вызывает `join_queue.add_task()` для добавления задач в очередь выполнения
   - Код создает задачи: `await self.ingestion.enqueue_join_tasks(distribution)`
   - Но не добавляет их в очередь: отсутствует вызов `await self.join_queue.add_task(...)`

2. **Загрузка старых задач при перезапуске**: Метод `load_pending_tasks()` в `JoinQueue` загружает ВСЕ задачи со статусом 'pending' из базы, включая те, которые были созданы давно и не были добавлены в очередь
   - При перезапуске системы старые задачи выглядят как задачи с огромными задержками

3. **Неправильное отображение в check_status.py**: Утилита показывает "через сколько времени выполнится задача" вместо "с какой задержкой была создана задача"
   - Формула: `time_diff = (scheduled_at - now)` показывает оставшееся время, а не изначальную задержку
   - Для старых задач это дает отрицательные или очень большие числа

4. **Возможная проблема с timezone**: Смешивание naive и aware datetime объектов может приводить к неправильным вычислениям временных интервалов
   - `datetime.now()` без timezone vs `datetime.fromisoformat()` с timezone

## Свойства корректности

Property 1: Bug Condition - Задачи добавляются в очередь после создания

_Для любого_ события создания задачи, где задача успешно создана в базе данных через `enqueue_join_tasks()`, исправленная функция `_process_pending_chats()` ДОЛЖНА добавить эту задачу в `join_queue` с правильными параметрами (task_id, userbot_id, chat_id, scheduled_at), чтобы задача была выполнена в запланированное время.

**Валидирует: Требования 2.1, 2.2, 2.4**

Property 2: Preservation - Создание задач с правильными задержками

_Для любого_ вызова `enqueue_join_tasks()` с параметрами join_delay_min и join_delay_max, исправленный код ДОЛЖЕН создавать задачи с задержками в диапазоне [join_delay_min, join_delay_max] секунд, сохраняя существующее поведение генерации случайных задержек для каждой задачи независимо.

**Валидирует: Требования 3.1, 3.2**

## Реализация исправления

### Требуемые изменения

Предполагая, что наш анализ корневой причины верен:

**Файл**: `main.py`

**Функция**: `_process_pending_chats()`

**Конкретные изменения**:
1. **Добавление задач в очередь после создания**: После вызова `enqueue_join_tasks()` добавить код для извлечения созданных задач из базы и добавления их в `join_queue`
   - Для каждой пары (userbot_id, chat_id) из distribution
   - Выполнить SELECT для получения task_id и scheduled_at только что созданной задачи
   - Вызвать `await self.join_queue.add_task(task_id, userbot_id, chat_id, scheduled_at)`

2. **Обеспечение timezone-aware datetime**: Убедиться, что все datetime объекты используют timezone.utc
   - При парсинге scheduled_at из базы использовать `.replace(tzinfo=timezone.utc)` если необходимо
   - Использовать `datetime.now(timezone.utc)` вместо `datetime.now()`

3. **Логирование для отладки**: Добавить логирование количества добавленных задач в очередь
   - `print(f"✓ Added {count} tasks to join queue")`

**Файл**: `check_status.py`

**Конкретные изменения**:
1. **Отображение реальной задержки**: Вычислять delay как `(scheduled_at - created_at)` вместо `(scheduled_at - now)`
   - Добавить извлечение created_at из базы данных
   - Показывать: "Delay: X seconds (Y minutes)" - изначальная задержка
   - Показывать: "Executes in: X minutes" - оставшееся время до выполнения

2. **Обработка просроченных задач**: Если `scheduled_at < now`, показывать "OVERDUE by: X minutes"
   - Это поможет идентифицировать старые задачи, которые не были выполнены

3. **Timezone-aware сравнения**: Использовать timezone-aware datetime для всех сравнений
   - `now = datetime.now(timezone.utc)`
   - При парсинге из базы добавлять timezone если отсутствует

**Файл**: `src/ingestion/join_queue.py` (опционально)

**Конкретные изменения**:
1. **Фильтрация старых задач при загрузке**: В методе `load_pending_tasks()` добавить фильтр для игнорирования задач старше определенного порога (например, 1 час)
   - `WHERE status = 'pending' AND created_at > datetime('now', '-1 hour')`
   - Это предотвратит загрузку накопившихся старых задач при перезапуске

2. **Автоматическая очистка просроченных задач**: Добавить метод для пометки просроченных задач как 'failed'
   - Задачи, у которых `scheduled_at < now - threshold`, помечать как 'failed'

## Стратегия тестирования

### Подход к валидации

Стратегия тестирования следует двухфазному подходу: сначала выявить контрпримеры, демонстрирующие баг на неисправленном коде, затем проверить, что исправление работает корректно и сохраняет существующее поведение.

### Исследовательская проверка условия бага

**Цель**: Выявить контрпримеры, демонстрирующие баг ДО реализации исправления. Подтвердить или опровергнуть анализ корневой причины. Если опровергнем, потребуется пересмотр гипотезы.

**План тестирования**: Написать тесты, которые симулируют добавление чата через дашборд и проверяют, что задача добавляется в очередь. Запустить эти тесты на НЕИСПРАВЛЕННОМ коде, чтобы наблюдать сбои и понять корневую причину.

**Тестовые случаи**:
1. **Тест добавления задачи в очередь**: Создать pending чат → вызвать `_process_pending_chats()` → проверить, что задача создана в базе И добавлена в `join_queue` (провалится на неисправленном коде)
2. **Тест задержки задачи**: Создать задачу → проверить, что `(scheduled_at - created_at)` находится в диапазоне [join_delay_min, join_delay_max] (провалится, если задержка 420+ минут)
3. **Тест множественных чатов**: Создать 5 pending чатов → вызвать `_process_pending_chats()` → проверить, что все 5 задач добавлены в очередь (провалится на неисправленном коде)
4. **Тест перезапуска системы**: Создать задачу, не добавив в очередь → перезапустить систему → проверить, что `load_pending_tasks()` загружает задачу (может провалиться, если задача слишком старая)

**Ожидаемые контрпримеры**:
- Задачи создаются в базе данных, но `join_queue.size()` остается 0
- Возможные причины: отсутствие вызова `add_task()`, исключение при добавлении, неправильные параметры

### Проверка исправления

**Цель**: Проверить, что для всех входных данных, где выполняется условие бага, исправленная функция производит ожидаемое поведение.

**Псевдокод:**
```
FOR ALL task_creation_event WHERE isBugCondition(task_creation_event) DO
  result := _process_pending_chats_fixed(task_creation_event)
  ASSERT task_added_to_queue(result.join_queue)
  ASSERT task_delay_in_range(result.task, join_delay_min, join_delay_max)
END FOR
```

### Проверка сохранения

**Цель**: Проверить, что для всех входных данных, где условие бага НЕ выполняется, исправленная функция производит тот же результат, что и оригинальная функция.

**Псевдокод:**
```
FOR ALL task_creation_event WHERE NOT isBugCondition(task_creation_event) DO
  ASSERT _process_pending_chats_original(task_creation_event) = _process_pending_chats_fixed(task_creation_event)
END FOR
```

**Подход к тестированию**: Property-based тестирование рекомендуется для проверки сохранения, потому что:
- Оно автоматически генерирует множество тестовых случаев по всему входному домену
- Оно выявляет граничные случаи, которые ручные unit-тесты могут пропустить
- Оно предоставляет сильные гарантии, что поведение не изменилось для всех не-багованных входных данных

**План тестирования**: Наблюдать поведение на НЕИСПРАВЛЕННОМ коде для не-багованных входных данных (например, задачи, созданные через другие пути), затем написать property-based тесты, фиксирующие это поведение.

**Тестовые случаи**:
1. **Сохранение создания задач**: Наблюдать, что `enqueue_join_tasks()` создает задачи с правильными задержками на неисправленном коде, затем написать тест для проверки, что это продолжает работать после исправления
2. **Сохранение распределения**: Наблюдать, что `distribute_chats()` использует round-robin на неисправленном коде, затем написать тест для проверки, что это продолжает работать после исправления
3. **Сохранение выполнения задач**: Наблюдать, что задачи, добавленные в очередь, выполняются в запланированное время на неисправленном коде, затем написать тест для проверки, что это продолжает работать после исправления
4. **Сохранение обновления статусов**: Наблюдать, что статусы задач обновляются корректно на неисправленном коде, затем написать тест для проверки, что это продолжает работать после исправления

### Unit-тесты

- Тест создания задачи и добавления в очередь для одного чата
- Тест создания задач и добавления в очередь для множественных чатов
- Тест обработки случая, когда нет доступных юзерботов
- Тест обработки случая, когда нет pending чатов
- Тест правильного вычисления задержки в check_status.py
- Тест отображения просроченных задач в check_status.py

### Property-based тесты

- Генерировать случайные конфигурации (join_delay_min, join_delay_max) и проверять, что все созданные задачи имеют задержки в указанном диапазоне
- Генерировать случайные распределения чатов и проверять, что все задачи добавляются в очередь
- Генерировать случайные состояния системы и проверять, что поведение распределения и выполнения задач сохраняется
- Тестировать, что для любого количества pending чатов (1-100) все задачи создаются и добавляются в очередь

### Интеграционные тесты

- Тест полного потока: добавление чата через дашборд → создание задачи → добавление в очередь → выполнение задачи
- Тест переключения между контекстами: добавление чатов, перезапуск системы, проверка, что задачи загружаются и выполняются
- Тест визуальной обратной связи: запуск check_status.bat и проверка, что отображаются правильные задержки и статусы
