# Документ требований к исправлению бага

## Введение

Система создает задачи присоединения к чатам с задержками 420+ минут (7+ часов) вместо настроенных 60-104 секунд. Несмотря на то, что файл config.json содержит правильные значения (join_delay_min: 60, join_delay_max: 104), задачи планируются с огромными задержками, которые примерно в 420 раз больше ожидаемых.

Этот баг вызывает серьезные операционные проблемы:
- Чаты не присоединяются вовремя (часы вместо секунд)
- Система выглядит нерабочей для пользователей
- Мониторинг лидов задерживается на часы, упуская срочные возможности

Баг сохраняется даже после многократной очистки базы данных, что говорит о том, что причина не в старых данных. Локальные тесты показывают правильное поведение (задержки 60-104 секунды), но продакшн сервер создает задачи с задержками 420+ минут. Это указывает на проблему конфигурации или инициализации, специфичную для продакшн окружения.

## Анализ бага

### Текущее поведение (Дефект)

1.1 КОГДА система создает задачи присоединения на продакшн сервере ТО задачи планируются с задержками 420+ минут (25,200+ секунд) вместо настроенных 60-104 секунд

1.2 КОГДА вызывается IngestionModule.enqueue_join_tasks() с join_delay_min=60 и join_delay_max=104 ТО результирующие задачи в базе данных имеют временные метки scheduled_at на 420+ минут в будущем

1.3 КОГДА config.json содержит join_delay_min: 60 и join_delay_max: 104 ТО система все равно создает задачи с задержками примерно в 420 раз больше максимального настроенного значения

1.4 КОГДА пользователь запускает clear_pending_tasks.bat и система пересоздает задачи ТО новые созданные задачи все равно имеют задержки 420+ минут

1.5 КОГДА debug_task_creation.bat создает задачи #1, #2, #3 с правильными задержками (9, 19, 34 секунды) ТО check_status.bat показывает Task #9 с задержкой 489 минут, что предполагает множественные пути кода или экземпляры

### Ожидаемое поведение (Правильное)

2.1 КОГДА система создает задачи присоединения с join_delay_min=60 и join_delay_max=104 ТО задачи ДОЛЖНЫ планироваться с задержками между 60 и 104 секундами (включительно)

2.2 КОГДА IngestionModule инициализируется параметрами из config.json ТО он ДОЛЖЕН использовать эти точные значения для всех операций создания задач

2.3 КОГДА загружается config.json ТО все экземпляры IngestionModule ДОЛЖНЫ получать и использовать правильные значения join_delay_min и join_delay_max

2.4 КОГДА задачи создаются через любой путь кода (enqueue_join_tasks, dashboard или другой) ТО они все ДОЛЖНЫ использовать одинаковые значения конфигурации

2.5 КОГДА система вычисляет scheduled_at = now + timedelta(seconds=delay_seconds) ТО delay_seconds ДОЛЖЕН быть значением между join_delay_min и join_delay_max

### Неизменное поведение (Предотвращение регрессии)

3.1 КОГДА задачи создаются с правильными значениями задержки (60-104 секунды) ТО система ДОЛЖНА ПРОДОЛЖАТЬ планировать их правильно

3.2 КОГДА IngestionModule.distribute_chats() назначает чаты юзерботам ТО он ДОЛЖЕН ПРОДОЛЖАТЬ использовать round-robin балансировку нагрузки

3.3 КОГДА задачи присоединения обрабатываются очередью присоединения ТО они ДОЛЖНЫ ПРОДОЛЖАТЬ выполняться в запланированное время

3.4 КОГДА файл config.json изменяется ТО система ДОЛЖНА ПРОДОЛЖАТЬ правильно перезагружать конфигурацию

3.5 КОГДА доступно несколько юзерботов ТО система ДОЛЖНА ПРОДОЛЖАТЬ равномерно распределять задачи между ними

3.6 КОГДА юзербот достигает дневного лимита присоединений ТО система ДОЛЖНА ПРОДОЛЖАТЬ исключать его из распределения задач

3.7 КОГДА задачи помечаются как завершенные или проваленные ТО база данных ДОЛЖНА ПРОДОЛЖАТЬ правильно обновлять их статус
