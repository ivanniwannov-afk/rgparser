# Bugfix Requirements Document

## Introduction

Критическая архитектурная ошибка в системе управления очередями задач приводит к полной блокировке Event Loop при получении FloodWait с длительным таймером (например, 400 минут). Когда аккаунт получает FloodWait, отложенная задача помещается в приоритетную очередь, но метод `get_next_task()` блокирует весь Event Loop через `asyncio.wait_for()` на время задержки. В результате все новые задачи с короткими таймерами, которые добавляются в базу данных, физически игнорируются, так как очередь "спит" и не может обрабатывать новые события. Дополнительно, логирование FloodWait ошибок слишком тихое, что затрудняет диагностику проблемы.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN аккаунт получает FloodWait с задержкой 400 минут THEN метод `get_next_task()` вызывает `asyncio.wait_for(self._stop_event.wait(), timeout=delay)` и блокирует Event Loop на 400 минут

1.2 WHEN Event Loop заблокирован ожиданием FloodWait задачи THEN новые задачи с короткими таймерами (например, 1-5 минут), добавленные в базу данных, не обрабатываются до истечения 400-минутного таймера

1.3 WHEN происходит FloodWait ошибка в `safe_join_chat()` THEN логируется только `logger.warning()` без яркого вывода в консоль, что затрудняет обнаружение проблемы

1.4 WHEN задача с FloodWait возвращается в очередь THEN она занимает верхнюю позицию в приоритетной очереди и блокирует обработку всех последующих задач

### Expected Behavior (Correct)

2.1 WHEN аккаунт получает FloodWait с любой задержкой THEN метод `get_next_task()` SHALL использовать событийно-управляемый механизм с `asyncio.wait()` для одновременного ожидания как таймера задачи, так и события новых задач

2.2 WHEN новая задача добавляется в очередь через `add_task()` THEN SHALL устанавливаться событие `_new_task_event`, которое немедленно пробуждает воркер для переоценки приоритетов

2.3 WHEN происходит FloodWait ошибка в `safe_join_chat()` THEN SHALL логироваться яркое предупреждение с использованием `logger.error()` и четким указанием времени блокировки

2.4 WHEN воркер пробуждается событием новой задачи THEN SHALL проверять всю очередь и выбирать задачу с наименьшим `scheduled_at`, даже если предыдущая задача еще не готова к выполнению

### Unchanged Behavior (Regression Prevention)

3.1 WHEN задача готова к выполнению (scheduled_at <= now) THEN система SHALL CONTINUE TO немедленно возвращать её для обработки

3.2 WHEN очередь пуста THEN метод `get_next_task()` SHALL CONTINUE TO блокироваться до появления новой задачи или сигнала остановки

3.3 WHEN задача помечается как completed/failed THEN система SHALL CONTINUE TO корректно обновлять статус в базе данных

3.4 WHEN система загружает pending задачи при старте THEN метод `load_pending_tasks()` SHALL CONTINUE TO восстанавливать все задачи из базы данных в правильном порядке

3.5 WHEN задача не готова к выполнению THEN она SHALL CONTINUE TO оставаться в очереди до наступления времени `scheduled_at`
